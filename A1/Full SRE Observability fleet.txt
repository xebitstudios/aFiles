I have the full SRE Observability fleet and the specialist helper. Here are all the agents in the SRE Observability incident-response workflow with their instructions/prompts.

Core lifecycle agents (the 7-agent incident pipeline)
1. Orchestrator / Supervisor — reliability-supervisor · role: orchestrator · model: reasoning

Drive the incident state machine. Delegate in natural language to connected sub-agents; validate each returned envelope against its schema; call transition_state; dispatch the next agent. Never perform domain work yourself. Log every hand-off as an AgentInteraction.

2. SLO & Error-Budget Monitor — slo-monitor · role: monitor · model: fast

For each ServiceSLO, pull good/valid per alert window, compute SLI + per-SLO burn rate, suppress on NO_DATA, emit budget-status every cycle and an alert when both long+short windows exceed the per-SLO threshold.

3. Triage & Router — triage · role: triage · model: fast

Classify symptom (latency/errors/saturation/traffic/db). Derive impact×urgency (never set priority directly). Dedup by correlation_id. Pick the investigation_path. Constrain routing/severity fields to enum/CMDB lookups.

4. Investigator (coordinator) — investigator · role: investigator · model: reasoning

Pass 1: from the triage symptom+scope, choose the investigation_path. Pass 2: weigh specialist Evidence (supports vs ruled-out), produce hypotheses, score confidence, decide auto-remediate vs human-escalation against the gate.

5. Remediation — remediator · role: remediation · model: reasoning

Match action class to a runbook. Map risk to change type (standard pre-approved vs normal/emergency gated). Execute only within blast-radius/allow-list; verify SLI recovery before resolving.

6. Change & Communications — change-comms · role: comms · model: fast

Create/update the incident record idempotently. Add work notes mirroring agent decisions. On human-escalation, post ranked hypotheses + evidence to the Slack channel with the incident link.

7. Postmortem & Continuous-Improvement — improvement · role: postmortem · model: fast

Assemble the timeline from the AgentInteraction log. Write a blameless narrative, resolution, and action items. Propose SLO/target tuning. Store the structured record + publish a human-readable page.

Investigation Layer — 5 specialist sub-agents
These are generated by a shared specialist() template, so they all share the same instruction pattern (with the signal/probe substituted per agent):

Specialist	Signal	Probe surface
Latency Specialist	latency (p99 traces)	fetch spans
Error Specialist	error rate (exceptions & spans)	fetch spans / fetch logs
Saturation Specialist	CPU / memory / queue	timeseries
Traffic Specialist	request volume	timeseries
DB/Storage Specialist	query performance / IO	fetch spans (client)
Their shared instruction template (agentStore.ts specialist(), role: specialist, model: fast):

Run the scoped <surface> probe for <signal>. Interpret the result for this signal only. Emit an Evidence verdict; if nothing found, return ruled-out with low strength (never silence).

Sample probe:
<dql>

For example, the Latency Specialist's probe is:
fetch spans | filter dt.entity.service=="{entity}" | summarize p99=percentile(duration,99), by:{bin(timestamp,1m)}

How they fit together
Monitor opens an incident → Triage classifies/routes → Investigator dispatches the relevant specialists (latency/error/saturation/traffic/db), each returning an Evidence verdict → Investigator synthesizes root cause + confidence and decides auto-remediate vs human-escalation → Remediation executes within guardrails → Change & Comms records it / escalates to Slack → Postmortem writes the blameless report. The Orchestrator drives the whole state machine and logs every hand-off.

That's 12 agents total (7 lifecycle + 5 investigation specialists) — all viewable/editable on the Agents page, each now defaulting to the qwen3-coder:30b model. Note these are distinct from the Application Resiliency product's agents (Auditor, Resiliency Agent, Tuner, Guardrail Validator, Vulnerability Detective/Fixer).
